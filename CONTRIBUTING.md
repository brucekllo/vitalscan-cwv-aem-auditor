# Contributing to VitalScan

This guide covers the three most common extensions: AEM detection rules, new metrics, and
CI thresholds. `shared/schema.ts` is the single source of truth shared by the frontend and
backend — start there for anything metric-related.

## Extend AEM detection rules

AEM signals are produced in `server/routes.ts` inside `runAudit()`. Each signal is an
`AemSignal` (`{ label, detected, detail }`) pushed onto `aemSignals`. The page is flagged as
AEM when at least one signal is `detected`.

To add a rule:

1. Open `server/routes.ts` and find the `// --- AEM detection layer ---` block.
2. Add a new check against `html`, `headers`, `scriptSrcs`, or `server`, then push a signal:

   ```ts
   const hasLaunch = /assets\.adobedtm\.com|launch-[A-Za-z0-9]+\.min\.js/i.test(html);
   aemSignals.push({
     label: "Adobe Experience Platform / Launch tag",
     detected: hasLaunch,
     detail: hasLaunch ? "Launch/AEP tag manager detected" : "Not found",
   });
   ```

3. The new signal automatically appears in the **AEM panel** (`client/src/components/Panels.tsx`)
   and the exported report — both iterate over `result.aemSignals`, so no UI change is needed.
4. To add advice copy, append to `AEM_TIPS` in `client/src/lib/recommendations.ts`.

Keep checks cheap (regex over already-fetched HTML/headers). Do not add extra network
requests inside the audit path — it must stay fast.

## Add a new metric

1. **Schema (`shared/schema.ts`):**
   - Add the key to the `MetricKey` union and to `METRIC_ORDER`.
   - Add an entry to `THRESHOLDS` with `good`, `poor`, `unit`, `source`, `sourceUrl`, and an
     optional `note`. Cite an official source (web.dev or Chrome for Developers).
   - If the metric should affect the overall score, add it to `LIGHTHOUSE_WEIGHTS` (weights
     are renormalized automatically — they do not need to sum to 1).
2. **Backend (`server/routes.ts`):** compute the value inside `runAudit()` and add it to the
   `rawValues` record. The rating/score/metric object is generated by the existing
   `METRIC_ORDER` loop, so no further wiring is needed.
3. **Frontend:** nothing required — `MetricCard`, the history table, trend charts, and the
   report all iterate over `METRIC_ORDER`. The new metric appears everywhere automatically.

`metricScore()` and `rate()` are threshold-driven and generic, so they work for any new
metric as long as **lower is better** (the app's invariant). If you need a higher-is-better
metric, add an inverted branch to those two functions and document it.

## Modify CI thresholds

Two files control CI gating:

1. **`lighthouserc.json`** — the source of truth for assertions. Edit the `assertions`
   block. Examples:

   ```json
   "largest-contentful-paint": ["error", { "maxNumericValue": 2200 }],
   "categories:performance": ["error", { "minScore": 0.85 }],
   "interactive": ["warn", { "maxNumericValue": 3500 }]
   ```

   Use `"error"` to fail the build, `"warn"` to report without failing. Audit IDs come from
   the [Lighthouse audit reference](https://github.com/GoogleChrome/lighthouse/blob/main/docs/scoring.md).

2. **`.github/workflows/lighthouse-ci.yml`** — the PR-comment table is rendered in the
   `actions/github-script` step. If you change a threshold in `lighthouserc.json`, update the
   matching row (value + threshold text) in the `body` array so the comment stays accurate.
   The job fails based on `lhci autorun`'s exit code, so the gate itself is driven entirely
   by `lighthouserc.json`.

## Local checks before opening a PR

```bash
npm run check     # TypeScript
npm run build     # production build must succeed
npm run dev       # manual smoke test at http://localhost:5000
```

## Conventions

- Keep `shared/schema.ts` as the single source of truth; never hardcode thresholds in the UI.
- All numeric values use the `.tnum` class (tabular numerals) so columns align.
- Status color comes from the `good` / `needs` / `poor` Tailwind tokens — never raw hex.
- Never rely on color alone: pair every rating with a label and icon (accessibility).
- The audit must remain a single server-side fetch — no per-asset network requests.
